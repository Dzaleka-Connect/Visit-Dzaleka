Authentication & Roles

This page describes how users log in to the Visit Dzaleka system, session lifetimes, role permissions, and how passwords and resets are managed.

Session Authentication

The application uses cookie-based session authentication. When a user logs in:

  • An encrypted session cookie is established in the browser containing the user's ID.
  • A CSRF token is generated and associated with the session. For any state-changing HTTP requests (POST, PUT, PATCH, DELETE), clients must supply this token in the X-CSRF-Token header.
  • By default, standard sessions are kept alive as long as active browser interaction occurs, matching the session middleware configuration.

User Roles & Authorization

Visit Dzaleka implements Role-Based Access Control (RBAC). Every account is assigned a single role which governs what pages and API endpoints they can access:

Role Description & Permissions
Admin Full administrative rights. Access to all dashboards, audit logs, email history, scheduled reports, webhooks, database configuration, user management, and pricing controls.
Coordinator Coordinates walking tours. Access to the master bookings list, guide assignments, calendar, and reports. Blocked from administrative tools (e.g. webhooks, API keys, audit logs).
Guide Tour guides operating in the camp. Restricted to the "My Tours" dashboard, availability settings, guide profile page, guide training materials, and payout details. Blocked from the bookings list.
Security Gate personnel. Restricted to check-in/out checkpoint tools, active visitor logs, incident reporting, and safety task boards. Blocked from billing, coordinator schedules, and master guides lists.
Transport Partner Transport providers. Restricted to the Transport Partner Portal, availability updates, quote bidding, pricing configurations, and their own profile.
Visitor General public visitors. Access to self-booking widgets, visit history, preparation resources, and customer support ticket submissions.

Password Management & Security

Password Hashing

All passwords are encrypted and hashed on the server using industry-standard hashing functions (scrypt/argon2 equivalent) before being stored in the database. Raw passwords are never logged or stored in plain text.

Changing Password

Users can change their password from the Profile/Settings page. Changing a password requires verifying the current password to ensure account ownership.

Password Reset Flow

If a user forgets their password:

  1. They navigate to the login screen and click Forgot Password.
  2. They enter their email address.
  3. The system generates a secure, single-use, cryptographically strong reset token that expires in 1 hour.
  4. An email is dispatched to the user containing the reset link: /reset-password?token=....
  5. The user opens the link, enters a new password, and submits. Once successful, the token is invalidated immediately.
